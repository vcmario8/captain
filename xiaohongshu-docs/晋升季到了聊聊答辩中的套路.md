# 晋升季到了，聊聊答辩中的套路

Leader 推荐你晋升，你要在规定时间内把项目讲清楚，评委听得懂是关键，评委听懂了项目，就能共鸣你了，一般就能通过了。

但让别人听得懂是比较难的，这是人性使然。分享做过的事情时天然就会按照时间顺序表达，就像播放已经存储的视频一样，一帧一帧过，因此和别人聊天时，大都能侃侃而谈；而学习，是需要把未知的知识录到大脑中，转变成一个个突触，大脑就会要求待吸收的东西必须经过结构化的整理，才能更快形成突触，这过程是反天性的。这也是为什么常说学习就是自讨苦吃。

回到答辩晋升场景，你讲做过的项目，就容易线性的表达；而面试官要理解你的项目，就要求项目必须是结构化的；因此要让面试官听懂，你就得结构化表达。这就是晋升套路。

## 具体怎么做呢？结构如下：

### 1. 现状

开门见山个讲3个问题，为什么是 3？因为人脑的堆栈深度就是 3，超过 4 就不太容易就记住了。每个问题要求讲清楚问题和背景。

### 2. 目标

针对每个问题，制定目标，阐述为什么定这个目标；是经过不停的比较、现状调研权衡来的，讲完就可以过了。不用太多停留。

### 3. 难点

2-3 个难点，定了这么高的目标，当然是不好解决的，提炼好难点，后面解决方案呼应难点。比如 100w 人同时点赞我的作品，赞计数有单点问题。

### 4. 业务特点

提炼 3 个业务特点，有什么用？是承上启下用的，用来后面做方案的权衡用的。比如100w 人同时对你作品点赞，你的作品赞计数，不需要严格的串行增加，只要数据往上累加即可；这就是点赞场景的特点，可以用来决策后面最终一致性的方案的。

### 5. 业界调研

调研 2-3 个业界实现，这一步在 晋升22（10 级）是加分项，晋升 31 （11 级）是必选项；每个方案都要点评下，它的方案解决的场景和你的业务特点有什么关系，最终得到结论，这些方案都不太能满足你的业务场景。（那肯定呀，你都提炼业务特点了，每个业务特点都是不一样的）

### 6. 解决方案

终于可以写方案了，千万别万精油的写最终一致性解决方案，应该是基于业务场景给解法，基于 xxx 和 xxx 的 xxx 解决方案，比如基于热 key 分区和最终一致性的赞技术解决方案。

### 7. 效果

带过就行，对比业界和目标。

### 8. 规划

效果再好也不能满足，要对下一步有规划；具体如何规划，篇幅原因，以后有机会再写。
